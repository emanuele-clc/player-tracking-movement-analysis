How it works

Seven stages, video in — insight out

Each stage is a real, independently-runnable script (see src/), validated on real footage rather than synthetic placeholders.

1

Detection & tracking

YOLOv8 finds every player/ball per frame; ByteTrack links detections into persistent tracks across occlusion.

2

Pitch calibration

Automatic line detection (HSV + Hough transform) feeds a homography — pixels → real-world metres, no manual clicking.

3

Team ID

K-means clustering on jersey colour splits every track into team A / team B / referee.

4

Tracking dataset

Detections + teams + pitch coordinates joined into one dataset, with real elapsed-time speed per track.

5

Movement analysis

Positional role clustering and per-player/team heatmaps from real tracked positions.

6

Original metric

A Voronoi-based off-ball space-creation score — see below.

7

Dashboard

This page — a single self-contained file, no backend, no database.

    Original contribution

    Off-ball space creation score

    The metric this project was built to produce — not a chart of where the ball was, but a number for what a player's movement did for their team. Computed on the same calibrated clip as the playback and heatmaps above.

    In plain terms: most tracking data can tell you where a player stood. This metric asks a sharper question — when a player moves, does it open up space for a teammate? Using every visible player's position each frame, a grid-based approximation of a Voronoi diagram assigns every patch of pitch to whichever player is closest to it. Summing a team's controlled patches, minus one player's own, gives that player's "teammate space." The score then correlates a player's own movement in a frame with how much that teammate space grows over the next couple of seconds — rewarding exactly the kind of intelligent decoy run or space-clearing movement that a heatmap or touch map cannot see, because those only show where a player was, never what their movement created for someone else.
    Honest caveat, shown rather than hidden: the method itself is verified correct — a synthetic 4-players-in-4-quadrants self-test gives 0.00% error against the exact expected area split (run python src/space_creation.py --self-test yourself). But the scores below, computed on the current real test clip, are numerically unstable — some values exceed 10,000 m² per metre of displacement. That's an honest symptom of too little data (only 13 of 114 frames have both teams visible at once in this box-focused clip), not a bug in the method. A meaningful, stable player ranking needs a full match's worth of tracking data.

    Technical validation

    How we know this is real, not a demo trick

    Every number below comes from cross-checking the pipeline against known, fixed facts about a real football pitch — not from trusting the model output blindly. This is what makes the calibrated clip used in the sections above trustworthy enough to report real metres and speeds from.

    4 / 4
    independent px/m scale estimates agree within 1.4%
    ~8cm
    mean reprojection error (12.6cm max)
    1
    real measurement bug caught & fixed by this validation
    14 / 18
    plausible real team split from jersey-colour clustering

    The pitch-calibration homography is checked against four independent real-world measurements — goal-box width/depth and penalty-box width/depth — computed from the same fitted scale, entirely algorithmically (src/auto_calibrate.py), not eyeballed.

    💡
    What this cross-check actually caught: the four scale estimates above didn't agree at first. Chasing that 15%+ disagreement down led straight to a real bug in the pitch-geometry constants — the penalty-box depth was hardcoded as 20.15m instead of the correct FIFA measurement of 16.5m. Once fixed, all four estimates converged to within 0.8-1.4% of each other and the homography's reprojection error dropped to ~8cm mean. The validation didn't just confirm the method works — it found and fixed an actual mistake before it could quietly corrupt every downstream metric.

    Try it yourself

    Run this on your own match

    This page is a fixed showcase of one validated clip. GitHub Pages is static, so it can't run the detection model on a video you upload — but anyone can download this project and run the full pipeline locally, on their own machine, on any footage they have. No GitHub account needed for this part.

    A local app (app.py) wraps the entire pipeline: upload a clip, click Analyze, get a full report — tracked video, team clustering, heatmaps, role clustering, and the space-creation score — generated from your own footage, saved on your machine, nothing uploaded anywhere.

    # download the code (no account needed - it's a public repo)
    git clone https://github.com/emanuele-clc/player-tracking-movement-analysis
    # or just download the ZIP from the green "Code" button on GitHub
    cd player-tracking-movement-analysis
    pip install -r requirements.txt
    streamlit run app.py

    # opens a local page in your browser -> upload a clip -> Analyze

    Pitch calibration in that flow is fully automatic and self-reports a confidence score (src/auto_calibrate.py) — on clips where it isn't confident enough, it honestly falls back to pixel-space tracking instead of guessing. Full reproduction steps in the README.
